API v1 — TO BE (заказы и списания)
Дата: 2026-03-16
Контур: staging -> prod

Цель
- Ввести единый и идемпотентный pipeline обработки заказов из витрин (InSales и будущие каналы).
- Автоматически списывать остатки в МойСклад только для оплаченных заказов.
- Хранить в AURA полный след: входящее событие, состояние заказа, результат списания, ошибки и ретраи.

Принципы
1) Source-of-truth по событиям заказа = витрина (InSales).
2) Source-of-truth по остаткам = МойСклад.
3) AURA = оркестратор и журнал факта обработки.
4) Exactly-once на уровне бизнес-действия достигается через idempotency + unique constraints.

Бизнес-правила
1) Оплаченный заказ
- При событии перехода в paid:
  - создать/обновить заказ в AURA,
  - запустить авто-списание в МС,
  - комментарий в write-off: "Продажа в InSales — заказ <номер>".
- Повтор такого же события не должен повторно списывать.

2) Неоплаченный заказ / корзина
- На v1 не делаем финальное списание.
- Для масштаба на другие витрины вводим слой резервов (phase 2):
  - reserve on pending/authorized,
  - commit on paid,
  - release on cancelled/expired,
  - TTL для резервов.

3) Частичная недоступность
- Если МС недоступен/ошибка транзитная -> retry с backoff.
- Если ошибка бизнес-валидации -> failed, ручная переработка из админки.

Событийный контракт (inbound)
Endpoint: POST /api/v1/insales/inbound/orders/events
Auth:
- HMAC подпись обязательна.
- Anti-replay окно: 300 секунд.

Минимальный payload (нормализуемый)
- source_event_id (обяз.)
- source: "insales"
- order.id, order.number
- order.status
- order.payment_status
- order.updated_at
- order.items[]: sku, quantity, price_minor, currency
- raw payload сохраняем целиком

Статусы (каноника)
Order lifecycle:
- received
- normalized
- awaiting_payment
- paid
- processing_fulfillment
- fulfilled
- failed

Fulfillment lifecycle:
- queued
- processing
- succeeded
- failed_retryable
- failed_terminal

Идемпотентность
1) Входящее событие
- unique(source, source_event_id)
- duplicate -> HTTP 200 + marker replay=true

2) Списание
- idempotency_key: "<source>:<external_order_id>:<payment_transition_version>"
- unique ключ в external_fulfillment_operations
- повторный запуск не создает второе списание

Интеграция с текущим write-off
- Используем существующий механизм списания из AURA (тот, что вызывается с /admin/product_stocks/:id/write_off).
- Добавляем сервис-обертку для пакетного списания из заказа.
- Для каждой строки заказа:
  - SKU -> Product
  - выбор склада по правилам канала (в v1: configured selling stores)
  - списать количество
- В write-off комментарий: "Продажа в InSales — заказ <номер>".

Админка (TO BE)
Раздел: Заказы InSales
- таблица заказов: номер, статус, оплата, сумма, updated_at
- статус обработки в AURA: awaiting_payment/fulfilled/failed
- колонка "Списание в МС": succeeded/failed + ссылка на документ/ID
- фильтры: номер, SKU, статус, период
- действие: "Повторить обработку" для failed_terminal

Known limitations (явно)
- v1 не обрабатывает резервы корзины (phase 2).
- v1 ориентирован на paid-триггер списания.
- multi-channel conflict resolution (одновременные продажи в разных каналах) — отдельный этап.

Критерии приемки v1
1) Оплаченный заказ из InSales приводит к одному и только одному списанию в МС.
2) Повтор webhook не создает дублей списания.
3) Ошибки МС не теряют заказ: есть retry/failed + видимость в админке.
4) В админке можно сверить заказ -> факт списания.

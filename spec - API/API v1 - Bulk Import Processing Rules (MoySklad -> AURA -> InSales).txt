API v1 — Правила обработки массовых импортов
MoySklad -> AURA -> InSales
Версия: v1
Дата: 15.03.2026
Статус: Draft (для внедрения)

==================================================
1) Цель
==================================================
Обеспечить стабильную работу сайта и интеграций при массовых импортах из МойСклад
(сотни/тысячи строк, повторные импорты, частые ошибки в файлах), без "пролегания" витрины.

Ключевой принцип:
- webhook intake всегда быстрый;
- тяжелая обработка только в управляемом фоне;
- last-write-wins;
- приоритет критичных изменений (особенно out-of-stock).

==================================================
2) Проблемы, которые закрываем
==================================================
2.1 Всплески вебхуков при пакетных загрузках из Excel.
2.2 Повторные импорты одного и того же файла (из-за правок менеджера).
2.3 Большой % строк без реального изменения данных.
2.4 Некачественные строки (часть импорта с ошибками).
2.5 Риск перегрузки AURA и InSales API.

==================================================
3) Обязательная архитектура обработки
==================================================
3.1 Прием событий (ingest)
- Вебхук НЕ запускает синк в витрину напрямую.
- Вебхук только пишет/обновляет запись в буфере событий (дедуп по product_id / sku).
- Ответ вебхуку сразу 200.

3.2 Буфер событий
- Одна активная запись на товар.
- Повторные события не дублируются, а коалессятся:
  - event_updated_at = max(old, new)
  - приоритет только повышается (normal -> high), не понижается
  - статус -> pending

3.3 Процессор батчей
- Отдельный worker читает из буфера батчами.
- Порядок чтения:
  1) high priority
  2) normal priority
- Параллельность безопасная: lock + skip locked.

3.4 Последовательность по товару (обязательно)
Для каждого товара строго:
1) пересчет каталога (InsalesCatalogItem)
2) проверка актуальности версии
3) sync в InSales

3.5 Защита от устаревших событий
- Перед sync повторно проверяем, что не пришла более новая версия события.
- Если пришла — текущий прогон пропускается как stale.

==================================================
4) Приоритизация событий
==================================================
4.1 high priority (обрабатывать в первую очередь)
- stock <= 0
- изменение по складу, участвующему в продаже
- изменение параметра, влияющего на цену/остаток в витрине

4.2 normal priority
- остальные изменения

==================================================
5) Политика нагрузки и стабильности
==================================================
5.1 Backpressure (автоматическое торможение)
Если backlog превышает порог:
- уменьшаем параллелизм sync в InSales;
- увеличиваем размер локального batch (для БД), но снижаем RPS наружу;
- временно отключаем необязательные операции (например media verify), пока backlog не нормализуется.

5.2 Rate-limit aware outbound
- Ограничение RPS в InSales клиенте.
- Retry только на retryable ошибки: 429, 5xx, timeout.
- Экспоненциальный backoff (1m, 5m, 15m, 1h ... cap).
- Circuit breaker при массовых 429/5xx.

5.3 Non-retryable ошибки
- Ошибка фиксируется как failed у конкретного товара.
- Очередь продолжает обрабатывать остальные товары.

==================================================
6) Правила качества данных из МойСклад (gate)
==================================================
6.1 После каждого большого импорта МС формируется QC-сводка:
- processed_rows
- changed_rows
- skipped_rows
- error_rows

6.2 Если error_rows > порога (настраиваемого):
- full/массовый экспорт в витрину блокируется;
- разрешены только high-priority изменения (чтобы не держать OOS товар в продаже).

Рекомендуемый стартовый порог:
- абсолютный: error_rows > 50
- относительный: error_rows / processed_rows > 2%

==================================================
7) Что считаем "безопасным состоянием"
==================================================
SLO (стартовые):
- p95 время от webhook до sync (high): <= 2 мин
- p95 время от webhook до sync (normal): <= 15 мин
- 5xx rate по API v1: ~0
- retry queue growth: контролируемый (без бесконечного роста)

Алерты:
- pending_high > 100
- pending_normal > 5000
- p95 high > 5 мин
- failed events > 100 за 1 час

==================================================
8) Наблюдаемость (обязательно в Dashboard/Monitor)
==================================================
Показываем:
- pending_high
- pending_normal
- processing
- failed
- stale_skipped_count
- retry_count_1h
- 429_count_1h
- p95_end_to_end_latency
- p95_sync_duration
- last_successful_sync_at

Для каждого run:
- run_id, batch_id, source
- total / updated / unchanged / skipped / failed
- top error codes

==================================================
9) Поведение при повторных импортов одного файла
==================================================
Если менеджер 2-3 раза загружает один и тот же Excel:
- буфер не растет линейно по количеству вебхуков;
- остается одна версия события на товар;
- фактический outbound sync на товар делается по последней версии.

==================================================
10) Режимы запуска
==================================================
10.1 Normal mode
- текущие лимиты, обычный приоритет.

10.2 Bulk-safe mode (авто или вручную)
- включается при всплеске импорта:
  - ужесточение rate-limit в InSales
  - усиление приоритета high
  - отключение вторичных задач

==================================================
11) План внедрения
==================================================
Этап A (быстрый, low-risk)
1) Документировать и зафиксировать пороги/правила.
2) В дашборд добавить метрики backlog/retry/429/p95.
3) Включить алерты.

Этап B (core)
1) Буфер + дедуп + last-write-wins.
2) Двухуровневый процессор high/normal.
3) Версионная защита stale перед sync.

Этап C (операционный)
1) QC-gate после больших импортов МС.
2) Bulk-safe mode + runbook действий команды.
3) Регулярный разбор failed/error rows.

==================================================
12) Критерии приемки
==================================================
- При импорте 1000+ строк очередь не растет 1:1.
- Повторы одного файла не создают дубли событий по товару.
- OOS изменения (stock<=0) доходят в приоритетном порядке.
- Сайт и админка остаются отзывчивыми под нагрузкой.
- Витрина не получает устаревшие данные (last-write-wins соблюдается).
